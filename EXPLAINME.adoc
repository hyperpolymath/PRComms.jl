== Claim 1: Message Architecture with Pillar-Variant Consistency Model

From README:
____
Message Architecture: Define pillars, proof points, and audience-specific variants to ensure coherent brand voice.
____

**How it's proven:**
- **Core struct**: `MessagePillar` exported at line 55 from `src/types.jl`
- **Variants**: `AudienceVariant` type at line 55; implementation in `src/messaging.jl`
- **Generation function**: `create_pillar(...)` and `generate_variant(...)` at lines 56 (exported)
- **Critical path**: Message consistency enforced by variant generation engine in `src/messaging.jl` which applies pillar content to audience-specific templates
- **Storage**: Pillar definitions stored with proof points; variants derived deterministically (same pillar + same audience = same variant)

**Caveat:** The proof-point linking system (README mentions "proof points") is structurally present but validation logic is incomplete. Pillars can be created without validated proof points; no automated fact-checking currently implemented.

== Claim 2: Newsroom Workflow with Embargo Lifecycle and Approval Gates

From README:
____
Newsroom Workflow: Manage the draft, review, and embargo lifecycle of press releases. Governance: Every external message requires explicit sign-off logs. Immutable Logs: Audit trail for all publication and outreach actions.
____

**How it's proven:**
- **Lifecycle states**: `PressRelease` type at line 55 with state machine implemented in `src/newsroom.jl`
- **Workflow functions**: `draft_release(...)`, `review_release(...)`, `publish_release(...)` at line 57
- **Critical path**: `src/newsroom.jl` (8KB) implements embargo logic with date-time constraints; release transitions from DRAFT → REVIEW → SCHEDULED → PUBLISHED
- **Audit trail**: Every status transition logs time, approver identity, and reason (immutable in design)
- **Approval gates**: `review_release(...)` enforces sign-off before publishing (exported at line 57)
- **Media contacts**: `MediaContact` and `PitchRecord` types at line 55 enable contact-history tracking

**Caveat:** Immutable log persistence is designed but not yet hooked to an audit database. Logs currently exist in-memory; server shutdown loses history. Planned: integration with `Lithoglyph` or `VeriSimDB` for permanent audit trail storage.

PRComms.jl is the **messaging tier** for ecosystem operations:
- **TradeUnionist.jl**: Uses PRComms for union press releases and member alerts (via `comms_bridge.jl`)
- **PostDisciplinary.jl**: May use PRComms for research dissemination and impact communication

== Critical Invariants (What Must Not Break)

1. Pillars must be immutable once published; variants generated from pillars, not modified directly
2. Release state transitions must be unidirectional: DRAFT → REVIEW → SCHEDULED → PUBLISHED (no backtracking)
3. All external messages (press releases, pitches, social posts) must be logged with timestamp and approver ID
4. Embargo dates must be enforced—no publication before scheduled time, even if manually triggered
